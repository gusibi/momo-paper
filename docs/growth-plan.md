# Momo Paper 30 天推广计划

## 目标

30 天内验证一件事：**是否有开发者愿意用 Momo Paper，让 Agent 生成第一份属于自己的正式文档。**

首月不以访问量或 star 为成功标准。核心指标是“首份自有文档渲染成功人数”，辅助指标依次是：内容详情访问、GitHub 访问、克隆/安装意向、示例渲染、真实文档渲染、7 日后再次使用。

## 首发定位

- **首要用户**：已经使用 Codex、Claude Code 或自研 Agent，并让它们产出研究报告、投资分析、产品方案的开发者。
- **首要场景**：把一段研究材料变成结构清晰、可打印的研究/投资报告。
- **用户痛点**：直接让 Agent 写 HTML/CSS，输出长、视觉漂移、修改困难；普通 Markdown 又不够正式。
- **产品承诺**：Agent 只写内容结构，Momo Paper 校验并渲染成统一样式的单文件 HTML/PDF。
- **首发主张**：`不要让 Agent 每次都从零手写正式文档的 HTML。`
- **唯一 CTA**：`拿一个真实任务试做第一份文档`。

首发文案不使用具体 token 节省比例。公开 benchmark 完成后，再把量化结果加入第二轮传播。

## 推广短板与补齐方式

| 短板 | 影响 | 本轮补齐 | 验收标准 |
| --- | --- | --- | --- |
| 用户范围过宽 | 文案像功能列表 | 锁定 Agent 开发者 + 正式报告 | 每条内容都能明确说出给谁、解决什么任务 |
| 缺少英雄案例 | 用户看不出实际价值 | 以 equity report 做一源多用案例 | 有需求、DSL、成品、命令、限制说明 |
| 文案只有标题 | 无法直接发布 | 建立 `promotion-copy.md` | 中文社区帖、长文、短帖、轮播、私信均有成稿 |
| 渠道没有节奏 | 容易一次性群发后失去反馈 | 建立 `content-calendar.md` | 每次发布间隔足够收集并修正反馈 |
| CTA 分散 | 用户不知道下一步 | 全部指向真实任务试用 | 每份内容只保留一个主 CTA |
| 缺少承接话术 | 评论和私信流失 | 增加评论回复与访谈邀请 | 常见异议能在 10 分钟内回复 |
| 缺少归因 | 无法判断渠道有效性 | 统一 UTM + 手工台账 | 每个外链能追溯 source/content |
| 缺可信量化证据 | “省 token”容易被质疑 | 用定性演示首发，benchmark 后量化 | 不发布没有原始数据的百分比 |

## 渠道分工

### 第一层：验证需求

- **GitHub**：产品事实、安装、示例和 issue 承接；所有渠道的最终依据。
- **V2EX 或掘金二选一**：验证中文开发者是否认可问题与工作流。首周不要两个都发。
- **一对一触达**：寻找 10 位正在做 Agent 报告输出的开发者，邀请用真实任务试用。它比泛流量更快暴露问题。

### 第二层：扩大触达

- **即刻 / X**：发布过程、前后对比和开发日志，引导到案例。
- **小红书**：用轮播展示“需求 → DSL → 成品”，触达 AI 工具与效率人群。
- **公众号 / 掘金**：承载完整技术文章和搜索流量。

### 第三层：英文社区

- **Show HN**：产品必须能直接试用，发布者本人在线回复。HN 禁止生成式或 AI 编辑文本，只能按本计划的事实提纲亲自撰写。
- **Reddit**：逐个 subreddit 核对规则；披露项目作者身份，以求反馈为主，不机械跨发。

## 英雄案例

首轮只做一套完整的 equity report 案例：

1. 输入：一份不含敏感数据的公司研究材料。
2. 任务：让 Agent 生成适合阅读和打印的投资研究报告。
3. 对照：普通 Markdown、Momo DSL、最终 HTML/PDF；没有合格 direct HTML 基线时不做量化比较。
4. 展示：20 行关键 DSL、整页成品、一次修改过程、validate 错误示例。
5. CTA：读者复制同一 prompt，或带自己的研究材料试用。

一套案例拆为：1 篇长文、1 个 README GIF、1 条社区主帖、6 张轮播、3 条短帖和 1 份可复制示例。不要同时生产多个浅案例。

## 内容支柱

每周内容只从四个支柱中选择：

1. **问题认知**：为什么正式文档不适合让 Agent 每次从零写 HTML。
2. **工作流演示**：自然语言任务 → DSL → validate → HTML/PDF。
3. **真实案例**：研究报告、产品 one-pager、复盘文档。
4. **构建过程**：错误、取舍、失败结果和用户反馈后的调整。

功能更新只有能落入上述内容支柱，或能解除安装/试用阻塞时，才进入首月排期。

## 指标与决策

### 每次发布记录

| 指标 | 含义 |
| --- | --- |
| 曝光 | 平台分发规模，只用于计算后续转化 |
| 内容详情/展开 | 标题和首屏是否有效 |
| 外链点击 | 内容是否产生进一步兴趣 |
| GitHub 访问 | 用户是否开始评估项目 |
| 安装/克隆反馈 | 使用意向，不等于激活 |
| 示例渲染成功 | 安装路径是否畅通 |
| 自有文档渲染成功 | 首月核心激活指标 |
| 有效对话 | 提供真实任务、问题或使用背景的回复 |

### 30 天阈值

- 至少触达 10 位目标用户进行一对一试用邀请。
- 至少 5 位成功生成自己的文档。
- 至少收集 10 条具体阻塞或反对意见。
- 至少 2 位在 7 天后再次使用或主动询问。

若内容有点击但无人完成示例，优先修安装和 Quick Start。若示例成功但无人生成自有文档，优先改案例与模板引导。若完全没有点击，先改问题定义和首屏，不开发新功能。

## UTM 规则

所有外链使用：

```text
utm_campaign=agent_docs_launch
utm_source=v2ex|juejin|jike|xiaohongshu|x|reddit
utm_medium=community|social|article|outreach
utm_content=<内容编号，例如 pain-01 或 demo-02>
```

GitHub 原始链接在不适合 UTM 的社区可直接使用；台账仍需记录发布日期、标题和帖子 URL。

## 外部账号操作清单

仓库内可以准备文案和素材，但以下动作必须由项目作者完成：

- 用本人语气改写并发布 HN 内容；发布当天保持在线。
- 发布前检查 V2EX 节点、Reddit subreddit 和小红书平台规则。
- 回复评论，明确自己是项目作者。
- 邀请真实开发者试用，不批量发送相同私信。
- 每周填写数据、整理原话，并决定下一周只修一个最大阻塞。
