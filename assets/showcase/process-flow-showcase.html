Momo Paper / web_dual.process_flow

流程图适合把 agent 的执行链路讲清楚。

这类页面最容易掉进“画得很复杂、看不出重点”的坑。系统化做法是只保留关键节点、明确输入输出、把主路径与异常处理分层，而不是把所有细节同时铺平。

Best Use
Pipeline / SOP
适合 agent 工作流、内容交付流程、业务操作链路和审批路径。
Flow Rule
4 步主链
核心链路建议控制在 4 到 6 个节点，异常分支另外展开。

一个适合 agent 的内容生成流程

Input -> Select -> Render -> Review
Step 01

接收任务输入

读取主题、surface、document shape、受众、数据形态和目标，先确定当前 route 是否被系统正式支持。

Step 02

匹配 route 规则

从白名单中锁定 `web_dual.process_flow`，继承 print-safe 约束、组件集合和固定 section recipe。

Step 03

填充固定结构

按 `lead -> core flow -> branch / variation -> operational takeaway -> meta` 写入标题、节点和说明。

Step 04

执行双重校验

检查颜色、字体、组件、图表和语气是否越界，并确认页面在浏览器和打印导出时都能保持结构清晰。

流程图里还可以展示什么

Variations
A / SOP

标准作业流

适合连续动作说明，例如“从需求到交付”的流程页。

B / Decision

分支决策流

适合多条件路径，例如是否需要图表、是否需要外部资料。

C / Ownership

角色分工流

适合展示谁负责输入、谁负责审校、谁负责最终发布。

D / Feedback

闭环反馈流

适合强调修改、复盘、训练和模板优化的迭代路径。
